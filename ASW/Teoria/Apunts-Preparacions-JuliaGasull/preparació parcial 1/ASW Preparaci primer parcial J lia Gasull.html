Apunts Tema 3. Web and Service Application Architectures

Introduction. Web Application Architectures. Service Application Architectures. Cloud Computing.

INTRODUCTION

▶ Software Architecture

Software architecture of a system is the set of structures needed to reason about the system, which comprise software elements, relations among them, and properties of both.

▶ Architectural Views

Logical view

is the large-scale organisation of the software components into packages, subsystems and layers that logically separate the functionality of a software system. It is called logical architecture because there is no decision about how these elements are deployed across different computational nodes.

Physical view

is the organisation and distribution of the logical architecture across different computational nodes in a network.

WEB APPLICATION ARCHITECTURES

▶ Logical - Physical Architecture

  • “Classic” Web Apps

  • Rich Internet Apps

▶ Dimensions to consider in a Physical Architecture Design

  1. Non-functional requirements that pursue the achievement of an adequate level of service
  1. Physical, financial and organisational constraints that may affect decision-making
  1. Alternative scenarios in architecture deployment

(1) Non-functional requirements

  • Performance: The application must sustain the expected workload defined in terms of
    • the maximum number of concurrent users,
    • the number of page requests served per unit of time,
    • the maximum time for delivering a page to the client
  • Scalability: The architecture must be easily extensible
  • Availability: Faults should not affect significantly the service delivered to users
  • Reliability (state maintenance): The state of the user interaction must be preserved, even when the application is distributed on multiple machines or failures occur
  • Security:
    • Data should be protected
    • Users should be identified and granted access only to the data and functions they are entitled to

(2) Constraints of Architecture Design

  • Cost
    • Every configuration requires a different investment, in terms of processors, network infrastructure, interfaces, and software licenses
    • The application budget may limit the choice of hardware resources and software products
  • Complexity
    • Some configurations are simpler than others to set up and maintain
    • The unavailability or the cost of specialised technical skills may constrain the architecture design
  • Corporate standards and infrastructures
    • The WebApp may be deployed within a corporate IT infrastructure, which may constrain the selection of hardware resources and software products.

(3) Scenarios of Architecture Deployment

  • Internal
    • The application architecture is kept inside the enterprise and maintained by the internal IT department.
  • Housed
    • The application architecture is maintained by the internal IT department of the enterprise, but is physically installed at an external service provider.
  • Hosted
    • The application architecture is located at the premises of an external service provider, who also maintains it.

▶ Architecture Patterns

Single server

  • Evaluation
    • Performance
      • Depends on the configuration of the server: CPU speed, available memory, disk access latency, etc.
      • The DBMS is both memory and CPU-intensive
    • Scalability
      • Is bound by the hardware architecture of the selected server
    • Availability
      • Every software and hardware element is a single point of failure: if it breaks, the entire system hangs.
      • Can be improved by adding redundant hardware resources (multiple CPUs, mirrored disks) and by installing multiple processes running different instances of the Web server, script engine, and database ...
    • State maintenance
      • No problems with none of the three possibilities to store user data: client, server or database
    • Security
      • A weak aspect of this configuration: attackers breaking the firewall and the Web server can take control of the host and gain direct access to the database, violating data protection
    • Low cost, as far as massive parallelism is not required.
    • Low complexity

Separate database

  • Evaluation
    • Better performance
      • One extra machine
      • Each tier can be tuned to the requirements of the installed software
    • More scalable
      • It is possible to act separately on each tier.
      • Normally, the first bottleneck is in the middle tier
    • Availability is not improved
      • Each component is still a single point of failure
    • Significantly improved security
      • The inner firewall may disallow HTTP requests at all and let only database requests pass, making it more difficult for attackers to reach the data tier

Replicated Web servers

  • Evaluation
    • Improved performance and scalability:
      • Load balancing
      • Clustering: A cluster is a group of servers (aka nodes) that provide a unified view of the services that they individually offer
    • Improved availability:
      • Fail-over: if a cluster node fails, its workload can be redistributed to the other nodes of the same cluster
    • Session state maintenance on the replicated servers
      • Session affinity (aka sticky sessions): The load balancer sends all the incoming requests pertinent to a given session to the same server.
      • Session migration: Session state is shared by the servers in the cluster
    • Improved data transmission security
      • One of the Web servers may be configured to handle the connections that require cryptographic protection.

Separated scripting engines

  • Evaluation
    • Improved performance, scalability and availability
      • Web server and the scripting engine can be replicated independently so that the number and configuration of the hosts can be optimised: a well-balanced configuration may require more machines for the scripting engines than for the Web servers.
      • The communication overhead introduced by the separation should be compensated by the performance increase.

Application servers

  • Evaluation
    • An application server is a software platform, distinct from the Web server, dedicated to the efficient execution of business components for supporting the construction of dynamic pages.
    • Improved performance, scalability and availability:
      • Transparent component distribution, replication, and load balancing:
        • The application server automatically manages the creation of processes, the replication of business objects and their allocation to the available processes, and the allotment of client requests to the increase and decrease of the actual workload.
      • Automatic failure recovery:
        • The application server may detect hardware, software and network failures, and avert client requests addressed to a failed component and route them to available replicas of the same business object.
      • Resource pooling:
        • The application server may handle pools of expensive resources, like database connections, and share these resource among multiple business objects in an optimised way

▶ Web Caching

Caching consists of temporarily storing resources in a fast access location, for later retrieval.

  • Benefits:
    • Reduction of the response time
    • Reduction of computation effort when the resource is dynamically build
  • Many things can be cached:
    • Static HTML pages and multimedia files.
    • Fragments of pages computed by scripting programs.
    • Intermediate data consumed by the scripting programs for producing page, e.g. XML files.
    • The result of database queries or other application commands

CLOUD COMPUTING

▶ 5 Essential Characteristics

  • On-demand self-service:

    A consumer can unilaterally provision computing capabilities as needed, such as server time and network storage, as needed automatically without requiring human interaction

  • Broad network access:

    Capabilities are available over the network and accessed through standard mechanisms.

  • Rapid elasticity:

    Capabilities can be rapidly and elastically provisioned.

  • Resource pooling:

    Physical and virtual resources are dynamically assigned and reassigned according to consumer demand.

  • Measured Service:

    Resource usage is monitored, controlled, and reported providing transparency for both the provider and consumer of the utilized service.

▶ Service Models

▶ Deployment Models

  • Private cloud:

    provisioned for exclusive use by a single organization

  • Community cloud:

    provisioned for exclusive use by a specific community of consumers from organizations that have shared concerns

  • Public cloud:

    provisioned for open use by the general public.

  • Hybrid cloud:

    composition of two or more distinct cloud infrastructures (private, community, or public) bound together by standardized or proprietary technology that enables data and application portability

P02_CSS_y_RWS

Cascading Style Sheets (CSS)

  • ¿Qué es CSS? - Cascading Style Sheets (Hojas de estilo en cascada)
    • Lenguaje de diseño gráfico que permite definir y especificar la presentación de elementos de una página web (definida con HTML o XML).
    • Podemos controlar la apariencia de una página web:
      • Estilo de fuentes.
      • Color y tamaño del texto.
      • Imagen o color de fondo,etc.
    • Tecnología usada para crear:
      • Páginas visualmente atractivas
      • Interfaces gráficas de usuario (GUI)
    • Uno de los pilares fundamentales de la Word Wide Web (WWW)
  • Historia
    • Origen (1970)
      • Popularidad de lenguajes de hojas de estilo → boom de Internet(70s).
      • Problema: estandarización - misma visualización en distintos navegadores
      • Organismo W3C propone un concurso para establecer un lenguaje de hojas de estilos específicos para HTML.
      • Las mejores propuestas:
    • Fechas importantes
      • CSS3 no es un paquete como las demás, es desarrollada por módulos
      • Cada nivel de CSS se basa en el anterior agregando nuevas características
  • Características
    • Separar el contenido del documento y la forma de presentación de este.
    • Un CSS se puede usar en distintas páginas web.
    • Box Model (cuadro que envuelve a cada elemento HTML)
    • Sintaxi de CSS
      • La declaración indica "qué hay que hacer"
      • El selector indica "a quién hay que hacérselo"
  • Tipos de selector - Los selectores CSS seleccionan elementos HTML de acuerdo con su id, clase, tipo, atributo, etc.
    • Selector elemento: Cuando el selector es un elemento HTML(<p>,<h1>, etc.)
      p { 
	color: red; 
	font-size: 20px; 
}
    • Selector ID: Cuando se define el selector usando el atributo ID de un elemento HTML. Se usa ”#” antes del id.
      <p id="demo1"> CSS sera aplicado aqui</p>
<p id="demo2"> CSS no sera aplicado aqui </p>
      #demo1 {
	color: red;
	font-size: 20px; 
}
    • Selector clase: Cuando se define el selector usando el atributo ID de un elemento HTML. Se usa ”.” antes del nombre.
      <p class="demo1"> CSS sera aplicado aqui</p>
<p class="demo2"> CSS no sera aplicado aqui </p>
      .demo1 {
	color: red;
	font-size: 20px; 
}
    • Selector universal: Se usa cuando se desea seleccionar y aplicar a todos los elementos de un HTML. Se ha de usar “*”.
      * {
	color: red;
	font-size: 20px; 
}
    • Selector de agrupamiento: Se usa para aplicar un mismo CSS en multiples elementos HTML.
      p, h1, h2 {
	color: red;
	font-size: 20px; 
}
  • Formas de agregar CSS
    • Estilo externo: el css se encuentra en un archivo externo del html, son connectados con la cabacera html. es el método más recomendado
    • Estilo interno: css va en la cabecera del html, dentro de una etiqueta style y su atributo type="text/css", desventaja para modificar, ya que se tiene que ir html file por html file
    • Estilo en línea: recomendado cuando se quieren corregir cosas específicas
  • Ejemplos
  • Puntos fuertes y débiles
    • ✅Ventajas
      • Ahorro del tiempo - un mismo css puede usarse en varias páginas
      • Carga rápida - menos código repetitivo en HTML
      • Mantenimiento sencillo - un sola modificación para efectuar un cambio global
      • Separa el contenido y de la presentación
      • Soporte en múltiples navegadores.
      • Permite que los sitios web se adapten a diferentes pantallas.
    • ❌ Inconvenientes
      • Confusión - debido a la existencia de varios niveles
      • Vulnerabilidad - “.css” al estar basado en texto, si alguien ajeno tiene permisos de escritura/lectura lo puede ser modificar
      • Podría haber problemas entre navegadores al usar CSS - un cambio puede ser expresado distinta manera por los navegadores

Responsive web design (RWD)

  • Qué es RWD y motivación
    • Enfoque de diseño web que pretende que las páginas que visualizamos lo hagan correctamente independientemente del dispositivo, orientación, pantalla y ventana
    • 51% del tráfico se produce desde smartphones y tablets
    • Mejor experiencia de usuario
    • Mejor posicionamiento SEO
    • Más facilidad y simpleza en el mantenimiento de la web
  • Historia
  • Características
    • Características principales
      • Responsive web design usa CSS y HTML
      • Hace uso de tres elementos básicos:
        • Viewports
        • Grid-views
        • Media Query
    • Viewports
      • El viewport es la parte que el usuario ve de una página web
      • HTML5 usa la etiqueta <meta> para indicar al navegador cómo controlar las dimensiones de la página y su escalado. El siguiente meta element debe incluirse en todas las páginas:
        • width=device-width → Indica que el ancho de la página sea el de la pantalla del dispositivo
        • initial-scale=1.0 → Define la relación entre la altura del dispositivo, y el tamaño del viewport.
    • Grid view
      • La mayoría de páginas se basan en una “grid-view”, esto quiere decir que dividen su página en columnas.
      • El uso de esta vista facilita el posicionamiento de los elementos. Normalmente se tienen 12 columnas.
      • Implementación:
        • Primero nos aseguramos que todos los elementos html tienen la propiedad box-sizing como border-box.
        • En nuestro CSS debemos añadir: *{ box-sizing: border-box; }
    • Media query
      • Técnica de CSS que se introdujo con CSS3
      • Utiliza la norma @media para incluir un bloque de css únicamente si una condición es cierta.
      • En este ejemplo si la ventana del navegador es de 600px o menos, el fondo será de color azul, de lo contrario será verde.
  • Puntos fuertes y débiles
    • ✅Ventajas
      • Mejor experiencia de usuario
      • Mantenimiento más sencillo
    • ❌ Inconvenientes
      • Mayor tiempo de carga de la web
      • Más tiempo al desarrollarse

P03_JavaScript_&_DOM

  • Introducció als continguts
    • JavaScript: Llenguatge de programació utilitzat principalment pel desenvolupament d’aplicacions web dinàmiques.
    • DOM (Domain Object Model): Interfície de programació per documents HTML/XML on es presenten els continguts en un estructura jeràrquica.
      • Cada element html es relaciona amb un node de l'arbre que genera DOM
      • És una api a partir de la qual els llenguatges de programació accedeixen als continguts html
    • Per què és important DOM? Facilita la comunicació entre llenguatges de programació i documents HTML.
  • Evolució històrica
    • Origen molt relacionat: necessitat de desenvolupar aplicacions web interactives.
    • El 1995 apareixen les primeres versions de JavaScript i DOM amb el navegador Netscape Navigator 2.0
    • Al 1997 la ECMA estandarditza Javascript.
      • Aquí es separen
        • DOM es generalitza
        • JS es continua desenvolupant com a llenguatge de programació
  • Característiques més rellevants
    • JS
      • Llenguatge interpretat
        • No es compila, s'interptreta el codi font (normalment amb el que ja porta el navegador)
      • Tipat dinàmic
        • Els tipus de la variable es defineix per el seu valor i no per com ha estat declarada
        • Un objecte pot tenir diferents tipus durant l'execució del programa, o que un array tingui diferents tipus d'elements
      • Llenguatge funcional
        • Les funcions són “ciutadans de primera classe”
        • funció = objectes (propietats, etc.)
      • Orientat a Objectes
        • Arrays associatius
          • nom: valor
        • Afegir, treure i iterar sobre les propietats dels objectes en temps d'execució
      • Basat en prototips
        • Herència basada en prototipus - propietat compartida per totes les instàncies d'una classe
        • haber estudiao cap
    • DOM
      • Permet cercar un element del document per id o tipus
      • Permet modificar el contingut HTML, els atributs o l’estil d’un element
      • Permet afegir múltiples listeners dels events d’un element.
        • Listener: funció que s'activa quan passa un determinat event a un element del document
      • Permet navegar i modificar la jerarquia de nodes d’un document HTML
        • Afegir o treure nodes, o navegar per jerarquies
  • Exemples il·lustratius
    • JS
      • Esquerra: tipatge dinàmic
      • Dreta: funció = objecte
    • DOM
  • ✅Punts forts
    • Client-side execution
      • Interpretat → execució ràpida
    • Alta Popularitat
      • Moltes fonts per aprendre'n
    • Interoperabilitat
      • Treballa amb altres llenguatges
    • Extended functionality
      • Llibreries de 3rs
  • ❌Punts dèbils / crítiques habituals (JS) (dom només podria millorar el debugger)
    • Facilitat en l’aparició de bugs i errors. → bugs silenciosos
    var i = 1;
i = i + ""; // Oops!
i + 1 → "11"
i-1→ 0
    • Problemes en la seguretat → s'executa en el navegador de l'usuari
    • Dependència del buscador.
    • Poc elegant i desestructurat: difícil d’escriure.
    • ”JavaScript fatigue” → evoluciona molt, difícil d'estar al dia

  • Conclusions i valoració personal
    • Dinàmiques → canvien internet
    • Crítiques a JS molt vàlides, es podrien corregir

    • Llenguatge de programació open-source.
    • Desenvolupat per Microsoft.
    • Presentat per primer cop en octubre de 2012.
    • Orientat a objectes.
    • Utilitzat per projectes grans.
    • Superset de JavaScript (qualsevol projecte de javascript funciona amb typescript (for real?))
  • Les fites
    • TypeScript 0.9:
      • variables genèriques (les que no son number, string o boolean)
      • nou compilador que resol més de 150 problemes que tenia l'antic
      • millora el rendiment al integrar-se en entorns de desenvolupament integrat → suposa un gran progrés per suportar codi de gran extensió
    • TypScript 1.0:
      • acceptar pull request de fora de microsoft
    • TypeScript 1.5:
      • compilador 5 vegades més ràpid que l'anterior
    • TypeScript 2.0:
      • capacitat d'evitar l'assignació de null a variables → evitar null pointer errors
        • com? amb dos tipus diferents de valors:
          • null
          • undefined
    • TypeScript 3.0:
      • millora de la manera en que surten els errors
    • TypeScript 3.4:
      • flag incremental (guarda compilació anterior → redueix temps de compilació)
  • ✅Característiques i punts forts
    • Utilització de tipus.
      • tipus de variables
      • tipus de retorn
      • millora comprensió
    • Compilació estàtica.
      • per identificar bugs en temps de compilació, no d'execució com amb javascript
    • Orientació a objectes.
      • classes tenen camps, constructors, propietats i funcions
    • És millor per projectes grans.
    • Compatible amb JavaScript.
    • Compatibilitat con els navegadors.
  • ❌Punts dèbils i/o crítiques habituals
    • Compilació lenta (TS → JS).
      • com que no és compatible amb cap navegador, es passa de TS a JS
    • JavaScript és més popular que TypeScript.
    • Corba de dificultat elevada. (1-2 mesos)
    • No admet classes abstractes però sí interfícies.
  • Exemples il·lustratius
  • Conclusions i valoració personal
    • Compilació estàtica i detecció d’errors en temps d’escriptura
      • trobem errors abans
    • Ús d’un llenguatge tipat
    • Orientat a objectes
    • Extensa compatibilitat
    • Inteligible per terceres persones

JSON

  • JSON
    • és una notació d'objectes JavaScript que s'usa per a poder representar i emmagatzemar estructures de dades en la sintaxi d'objectes de JavaScript.
    • sovint, s'utilitza per transmetre dades en aplicacions web, del client al servidor, o del servidor al client
  • Fites i història
    • Basat en un subconjunt de l’estàndard “JavaScript Programming Language Standard ECMA-262 3rd Edition – December 1999”
    • Douglas Crockford fou qui va especificar i popularitzar el format JSON
    • Abril 2001- Douglas Crockford envía el primer missatge JSON
    • Durant el 2005, la popularitat de JSON es dispara
      • Raons
        • Yahoo el va començar a utilitzar
        • (més important) Arribada d'aplicacions d'una sola pàgina i modernes apps mòbils
    • Octubre 2013 – Ecma International publica la primera edició de l’estàndard JSON ECMA-404
  • Característiques
    • Escrit en format parella: nom – valor
    • Defineix diferents tipus de valors
      • Simples: números, cadenes, booleans, nul
      • Complexes: valor, matrius i objectes
    • Independent del llenguatge
      • La seva sintaxi
        • fa que resulti senzill de llegir i escriure pels programadors
        • fa que sigui fàcil d'interpretar i generar per les màquines
    • Redueix el volum de les dades a transmetre
      • text/plain
      • fàcil de transformar a objecte JS amb la funció parse
    • Útil per: serveis web, API RESTs, aplicacions en JavaScript...
      • extensions del navegador
      • crides asíncrones
  • Exemples
    person = { "name":"John", "age":30, "car":null }; 
name_person = person.name;
name_person = person[“name”];
person[“name”] = “Jordi”;
    Person = {
	"name":"John", 
	"age":30, 
	"cars": {
		“car1”:”Ford”, “car2”:”BMW”, 
	}
	“identifiers“: [1,2,3] 
}
Car1 = person.cars[“car1”]; 
Delete person.cars.car2;
  • ✅Punts forts
    • Autodescriptiu, fàcil d’entendre i aprendre.
    • Flexible
      • suporta dos tipus d'estructures
        • parells clau, valor
        • arrays de valors
      • a qualsevol estructura se li poden afegir propietats fàcilment
    • Ràpid
      • disminueix el pes
    • Popular i segur
      • els navegadors xifren i dexifren JSON → més segur
    • Útil en serialització i adequat en l’intercanvi de dades
  • ❌Punts dèbils
    • No admet comentaris
    • No té definit el format exacte dels missatges
    • Limitat a l’emmagatzament de dades comunes
  • Conclusions i valoracions personals
    • Un format molt simple i útil en un ampli rang de tecnologies
    • Dels formats més utilitzats
    • Molt veloç
    • Encaixa perfectament en el tipus de serveis actuals

YAML

  • YAML Ain’t Markup Language
    • Objectiu: serialitzar dades en un document de text pla i llegible pels humans
    • Recent
    • S'utilitza per tasques on la lectura del contingut sigui important (e.g. arxius de configuració)
  • Fites i història
    • Ingydöt Net, Oren Ben-Kiki i Clrack Evans proposen el nou format YAML el 2001.
    • Desde 2006 l’extensió oficial de YAML ha estat .yaml
    • 2009 – Tercer llançament de YAML (actual)
    • El primer significat de l’acrònim fou: Yet Another Markup Language
      • ara: YAML Ain’t Markup Language
  • Característiques
    • Superconjunt de JSON
      • parser yaml pot entendre json, però no necessàriament al revés
      • yaml és un estàndard de serialització de dades → mapeja estructures de dades més complexes, com per exemple llistes
      • Compatible amb tots els llenguates de programació
      • S'utilitza principalment per a fitxers de configuració o depuració degut a la facilitat per entendre'l
    • Sintaxis senzilla
      • desenvolupar amb la creença que totes les estructures de dades es poden representar com a conjunt de llistes, hashes i valors simples
    • Estructura denotada indentant amb espais en blanc, no tabulacions
    • Utilitza UTF-8 i UTF-16
      • caràcters reservats per futures versions com
        • ,
        • *
        • @
    • Múltiples documents dins d’un únic flux
    • Requereix comes i punts i comes com a separadors

  • ❌Punts dèbils
    • Més lent per escriure estructures

P05_JSONP_&_CORS

  • Dos tecnologies que permeten demanar i comunicar de dades entre servidors de dominis diferents
  • Problemàtica
    • Intercanvi de diferents domini
    • Un client, per exemple el navegador firefox, fa una petició get a intranet.avengers.com i una petició get a gatitos.avengers.com. Si aquestes dues pàgines tenen el seu domini a un servidor extern, seltarà un error de:
      • "Cross-Origin Request Blocked: The Same Origin Policy disallows reading the remote resource a URL" → ERROR DE SEGURETAT
  • Història de JSONP
  • Què és JSONP?
    • Tecnologia que permet enviar dades JSON sense preocupar-se sobre *RIP Gramàtica*
    • JSON amb Padding
    • JSON encapsulat en un "Callback"
    • Incloure fitxer a <script></script>
  • Exemple JSONP
  • Història de CORS
  • Què és CORS?
    • Alternativa moderna a JSONP
    • És un mecanisme que permet que es puguin solicitar recursos d'origin creuat a una pàgina we
    • Per la nostra seguretat, els exploradors restringeixen la solicitud http d'un origin creuat, és a dir, segueixen la política de mateix origen. Això vol dir que només es poden fer solicituds http al seu propi domini, sempre i quan no utilitzis cors.
    • Cors dona controls d'accés de domini creuat i transferència segura de dades.
    • Els exploradors moderns utilitzen cors.
    • Petició d'accés al servidor de domini diferent
    • Flux de dades amb CORS
      • Quan una pàgina web vol connectar-se a un altre domini, el navegador agafa la petició i afegeix un request per tenir l'autorització.
      • Això li diuen pre-flight perquè consisteix en agafar un mètode httpOptions amb una sèrie de headers que informaran al servidor sobre les operacions que vol dur a terme.
      • El servidor rebrà la informació i retornarà una sèrie de headers dient-li que accepta la petició
      • El navegador té unes 5h on podrà fer les peticions acceptades pel servidor.
  • Exemple CORS
  • JSONP vs CORS

    Comparació

    ComparacióJSONPCORS
    UntitledNomés GETTots els mètodes
    UntitledTot tipus de navegadorsNavegadors moderns
    UntitledIncloure <script>Preflight request

    UntitledMenys segurMés segur
  • Conclusió
    • Cors s'utilitzaria en cas que tinguessis un navegador modern mentre que JSONP funcionaria amb un navegador més antic.

P09_XML

  • Definició
    • eXtensible, Markup, Language
    • Un llenguatge que determina una serie de lleis per codificar un document mitjançant etiquetes, de manera que codifiques el text i l'estructura a la vegada
    • Es tracta d'una eina que permet definir elements per tal de crear un format i unes etiquetes personalitzades
    • Més que un llenguatge, és una eina per definir el teu propi llenguatge → molt personalitzada
    • Es centra en la simplicitat i facilitat d'utilitzar-lo
  • Història
    • Prové de GML: general markup language (necessitat per emmagatzemar dades i exportar-les)
    • HTML tenia moltes etiquetes diferents interpretades per difetents visors → un lio, vamos
    • W3C buscava poder tenir un estàndard → desenvolupa XML
      • html
        • contingut es barreja amb estils
        • no parmet compartir amb diferents
        • presentació depen del visor
      • xml
        • permet barrejar elements de diferents llenguatxes i afegir noves etiquetes
        • analitzadors podien ser simples i estables
        • llavors dades eren fàcilment portables
  • Característiques
    • creació d'etiquetes pròpies i permet assignar atributs a aquestes etiquetes
    • estructura i disseny estan completament separats
    • s'emmagatzema en format text → estructura entenedora
    • cada document inclou metadates de sí mateix → ajuda amb la cerca dels navegadors
    • permet exportació a altres fomats (html, pdf)
    • estàndard obert, sense cap tipus de llicència
    • permet internacionalització → qualsevol caràcter
    • lleis de generalització concretes → documents fàcilment processables
    • permet compartir info entre pg web i bases de dades
    • molt portable, molt fàcil d'entendre, i facilita la feina a la gent
  • Code examples

  • ✅Avantatges

    • Facilita la comunicació entre aplicacions

P10_SOAP

  • Simple Object Acess Protocol
    • Protocolo de comunicación usado para intercambiar información estructurada en la implementación de servicios web y redes de computadores
    • Basado en XML para enviar y recibir mensajes y recomendado por el W3C (World Wide Web Consortium)
    • Usado principalmente en arquitecturas orientadas a servicios (SOA)
  • Por qué SOAP?
    • Comunicarse a través de internet es crucial para las aplicaciones web, la mejor manera de hacer esto es mediante HTTP, ya que está soportado por todos los navegadores.
    • SOAP fue creado para cumplir esto, provee una forma de comunicarse entre aplicaciones web que corren sobre sistemas operativos totalmente distintos, con diferentes tecnologías y diferentes lenguajes de programación.
    • Neutralidad: sobre TCP
  • Historia
  • Características relevantes
    • Extensibilidad: seguridad y WS-Routing son extensiones aplicadas en el desarrollo.
    • Neutralidad: bajo protocolo TCP puede ser utilizado sobre cualquier protocolo de aplicaciones como HTTP, SMTP o JMS.
    • Independencia: permite cualquier modelo y lenguaje de programación.
  • Funcionamiento
    • Un mensaje SOAP es un documento XML formado por las siguientes partes:
      • Envelope: raíz de la estructura, identifica el mensaje SOAP.
        • De uso obligatorio.
      • Header: permite enviar información de cómo debe ser procesado el mensaje.
        • Este elemento a su vez se compone de “Header Blocks”, que delimitan las unidades de información necesarias del header.
      • Body: contiene la información relativa a la llamada y la respuesta.
        • De uso obligatorio.
      • Fault: contiene información relativa a errores
  • Funcionamiento - ejemplo
    • Mensaje SOAP del cliente:
      <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
	<soap:Body>
		<getProductDetails xmlns="http://warehouse.example.com/ws">
			<productId>827635</productId>
		</getProductDetails>
	</soap:Body>
</soap:Envelope>
    • Respuesta del servidor:
      <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
	<soap:Body>
		<getProductDetailsResponse xmlns="http://warehouse.example.com/ws">
			<getProductDetailsResult>
				<productName>Toptimate 3-Piece Set</productName>
				<productId>827635</productId>
				<description>3-Piece luggage set. Black Polyester.</description>
				<price>96.50</price>
				<inStock>true</inStock>
			</getProductDetailsResult>
		</getProductDetailsResponse>
	</soap:Body>
</soap:Envelope>
  • Modelo de procesado
    • Soap está definido como un sistema distribuido, en el que intervienen distintos servicios (nodos). Hay 3 grandes tipos de nodos:
      • SOAP Sender: Nodos que transmite un mensaje SOAP.
        • Initial: Es el nodo inicial que origina el mensaje.
      • SOAP Receiver: Nodos que aceptan un mensaje SOAP.
        • Ultimate: Es el nodo destino del mensaje SOAP transmitido por el Initial SOAP Sender.
      • SOAP intermediary: actúa como sender y receiver a la vez, no realiza ninguna acción con el mensaje más que leer los headers para enviarlo al siguiente nodo.
  • ✅Pros
    • Permite la interoperabilidad entre múltiples entornos.
    • No está asociado con ningún lenguaje.
    • No se encuentra fuertemente asociado a otro protocolo de aplicación.
    • Aprovecha los estándares existentes en la industria.
  • ❌ Contras
    • Es más lento que otros protocolos de comunicación como CORBA, RMI o IIOP.
    • Depende de WSDL. No tiene descubrimiento dinámico de servicios.
    • La especificación no hace mención a las medidas de seguridad que utiliza.
    • Clientes no mantienen referencia al estado de los objetos remotos.
  • Conclusión
    • Estándar flexible que permite el itercambio de datos entre aplicaciones hechas con distintos lenguajes
    • Se utiliza en muchas grandes organizaciones, donde la seguridad es más primordial que su rendimiento
    • Su uso está decayendo en favor al principio RESTful, se usa en aplicaciones antiguas mayoritariamente (con la llegada de mongodb y rest)

P11_WSDL

  • Web Services Description Language
  • Què és i per a què serveix?
    • WSDL és un format del “Extensible Markup Language” (XML) que és utilitzat per a descriure la funcionalitat oferta per un servei web.
    • Descriu la forma de comunicació amb aquest servei, és a dir, com pot ser cridat el servei, quins paràmetres espera i quina estructura de dades retorna (en xml).
    • Especifica l'interfície pública a travésde la qual un client pot accedir a un servei web i els detalls de com s'ha d'utilitzar.
  • Versions (3)
    • Versió 1.0: Va ser la primera recomanació per part del W3C
    • Hi va haver la 1.1, sense molts canvis
    • Versió 2.0 (abans 1.2): Accepta el enllaç a tots els mètodes de solicitud http, ofereix un millor suport pels serveis webs restful, i és més simple d’implementar.
    • Diferència: (IBM, microsoft i ariba)
      • Es van agregar més semàntiques
      • Es van eliminar les construccions de missatges
      • La sobrecàrrega de l'operació ja no es suporta
      • Porta *rip gramatica* ja no es suporta
      • Ports renombrats a endpoints
  • Utilització
    1. El client sol·licita el servei web al fitxer WSDL
    1. El servidor entrega el fitxer WSDL demanat
    1. El client fa la petició en el format esperat pel servidor
    1. El servidor entrega el resultat de la consulta
  • Aplicació amb XML i SOAP
    • El programa client que es connecta a un servidor web, llegeix el WSDL per determinar quines funcions estàn disponibles en el servidor
      • El tipus de dades utilitzats es troben en el fitxer WSDL en format de XML Schema
      • El client pot fer servir SOAP per fer trucar alguna de funcions en el WSDL utilitzant per exemple XML en HTTP
  • Estructura
    • Tipus de dades
      • En aquesta secció es defineixen els tipus de dades utilitzats als missatges. S’utilitzen els tipus definits a l’especificació del XML Schema.
    • Missatges (1.0)
      • Aquí es defineixen els elements de missatge. Cada missatge pot consistir en una sèrie de parts lògiques. Les parts poden ser de qualsevol dels tipus definits en la secció anterior.
    • PortTypes (1.0) / Interfícies (2.0)
      • En aquest apartat es defineixen les operacions permeses i els missatges intercanviats en el servei.
    • Bindings
      • Aquesta secció defineix els protocols de comunicació utilitzats
    • Ports (1.0) / Endpoints (2.0)
      • Defineix l’adreça o punt de connexió al servei web.
    • Serveis
      • Set de funcions del sistema. Aquesta part final fa referència a tot allò aportat per les seccions anteriors.
  • ✅Punts Forts:
    • Fàcil interpretació entre els clients i el servei Web
    • Permet a les empreses comunicar-se detalladament sobre el funcionament del seu sistema
  • ❌Punts Dèbils:
    • Difìcil de comprendre per un humà
    • Cada cop que hi ha un canvi en el servidor s’ha de cambiar de fitxer WSDL
    • WSDL no permet especificar la seqüència de les operacions necessàries en un intercanvi de missatges.
  • Conclusions
    • WSDL permet descriure serveis web de manera formal
    • WSDL és molt flexible, ja que no es limita a una codificació, a un protocol de transport ni a un model d’invocació
    • WSDL és utilitzat en combinacions de SOAP i un XML Schema per a oferir Web Services a Internet
    • Facilita als clients saber que fa un Web Service, com ho fa i de quina manera pot interactuar amb ell
    • El programa client llegeix el WSDL per saber les operacions del servidor

P12_gRPC_ProtocolBuffers

  • Introducció
    • Avui en dia, les aplicacions web són un dels entorns més utilitzats en el camp de la informàtica.
    • Totes aquestes tasques són necessàries gràcies als microserveis, i aquests s'han de poder comunicar entre ells.
    • El problema és que aquesta comunicació és molt costosa. Un dels principals problemes és que poden estar en diferents llenguatges.
    • Comunicació entre mircroserveis complicada + Creació d’APIs costosa = gRPC + Proto Buffers
  • RPC
    • Crida de processament remot (Remote Procedure Call).
    • Molt utilitzat en comunicació client-servidor.
    • Executar codi en una màquina remota sense preocupació entre la comunicació entre el client i el servidor.
  • gRPC
    • gRPC és un protocol RPC open-source creat per Google.
    • gRPC Alfa i gRPC 1.0.
    • Està construït en HTTP/2, permet una comunicació bilateral. A més a més té autentificació per Tokens i una varietat de llenguatges de programació.
    • Comunicació entre microserveis òptima i simple.
    • Utilitza Proto Buffers i no JSON.
    • En alt nivell permet definir Request i Response per RPC, la resta és gestionat per gRPC.
  • Proto Buffers
    • Utilitzats per definir:
      • Missatges: Data, request i response.
      • Serveis
    • Es representen en binari.
  • ✅ Avantatges Proto Buffers
    • (binari) Payload molt inferior.
    • Menor ús de CPU.
    • Menor espai i major velocitat.
    • Suport en múltiples llenguatges.
  • Fites
    • 1981: RPC
    • 2001: Protocol Buffer (versió interna)
    • 2008: Protocol Buffer (versió pública)
    • 2015: gRPC Alfa (començat 2014)
    • 2016: Proto Buffer v3 i gRPC 1.0
  • Per què utilitzar gRPC i Proto Buffers
    • ✅ Rendiment:
      • Ràpid
      • Eficient
    • ✅Suport multillenguatge

P15_AsyncAPI

  • Introducción
    • Para empezar, vamos a definir lo que es una API.
    • Una API síncrona o asíncrona es una interfaz de programación de aplicaciones que devuelve datos respondiendo a solicitudes (requests), de forma inmediata o no immediata, respectivamente.
    • Las API síncronas o asíncronas proporcionan una forma de realizar solicitudes inmediatas o programadas de recursos, datos o servicios cuando estén disponibles.
    • La naturaleza síncrona y asíncrona de una API es una función del marco temporal desde la solicitud hasta la devolución de los datos.
    • En el caso de las API asincrónicas, la disponibilidad de un recurso, servicio o almacenamiento de datos puede no ser inmediata.
    • Estas API pueden proporcionar un callback al solicitante cuando el recurso solicitado esté listo.
    • Las solicitudes asincrónicas son útiles para mantener la funcionalidad en una aplicación en lugar de vincular los recursos de la aplicación que esperan una solicitud.
  • Historia
    • En 2017, Fran Méndez se encontraba trabajando con la API Changelog, una API reactiva o event-driven. Él y sus compañeros recomendaban a sus clientes que utilizaran OpenAPI para los mensajes síncronos, pero no tenían ninguna herramienta para aquellos eventos asíncronos, por lo que debía adaptar OpenAPI para estos casos. Después de observar que este problema era algo común entre los usuarios de OpenAPI decidió unirse con New Relic para integrar AsyncAPI.
    • AsyncAPI ha sido desde su nacimiento open source y esto ha contribuido al crecimiento de esta API. De este desarrollo podemos destacar cuatro versiones principales hasta la fecha, las versiones 1.0.0, 1.1.0, 1.2.0 y finalmente la versión actual 2.0.0.
    • En las imágenes de la derecha podemos observar dos esquemas que representan la estructura de OpenAPI y AsyncApi. Como se puede comprobar, las diferencias entre ambas API son pocas, pero podemos destacar principalmente el sector del mensaje que cambia el encabezado, la inserción de un identificador de la aplicación o la unión de los hosts y security.
  • Características
    • AsyncAPI es una arquitectura Event-Driven y por tanto sus características principales son propias de este tipo de APIs.
    • Como habíamos introducido anteriormente, este tipo de API es asíncrona y por lo tanto su funcionamiento no requiere de una respuesta inmediata. Como la mayoría de arquitecturas Event-Driven, funcionan con un sistema llamado Broker-Centric.
    • Un sistema Broker-Centric tiene tres componentes principales. El Publisher, el Broker y el Subscriber. El Publisher es la aplicación que se encarga de enviar el mensaje que quiere que reciban sus Subscriber. Éste le enviará el mensaje al Broker, una infraestructura encargada de almacenar y enviar los mensajes o eventos a los distintos Subscribers interesados. Y finalmente los Subscribers, que son otras aplicaciones, recogen los mensajes de los Brokers que quieran recibir.
    • AsyncAPI nació como adaptación de OpenAPI y por tanto comparte muchas de las características de esta API además de compatibilidad con los schemes de OpenAPI. Aún así existen diferencias relevantes entre estas además de la asincronía. El message payload en la AsyncAPI puede contener otros schemes además de OpenAPI o AsyncAPI, como por ejemplo un Avro schema. Además, AsyncAPI renombra lo que se entiende como schema en OpenAPI como protocols y amplía el concepto añadiendo la protocol version. Otra diferencia importante es la ausencia de queries o cookies, debido a la naturaleza asíncrona de la API.
    • Ejemplos ilustrativos y Funcionamiento interno
      • Aqui vemos un poco mas el funcionamiento interno de AsyncAPI
      • Siendo M2 el ToolKit proporcionado por AsyncAPI y usando su gramática, pasandole las especificaciones de la API (ya sea en YAML o en JSON) , vemos cómo se crea el modelo de la API y su metadata necesaria para la utilización de AsyncAPI y se generan los códigos y archivos en su DSL (Domain-Specific Language)
      • Workflow:
        • Aqui vemos el workflow de AsyncAPI que se basa en los siguientes pasos:
          • El cliente hace una petición asíncrona al servidor
          • AsyncAPI crea un “trabajo” para ese cliente y responder al cliente y le devuelve la ID de la petición (requestID)
          • Entonces el cliente va preguntando al servidor si el resultado está listo
          • AsyncAPI le responde o bien que no está completado aún, o si lo está, le devuelve el resultado
      • Puntos débiles y fuertes
        • AsyncAPI es una especificación diseñada específicamente para describir microservicios impulsados por eventos. Muchos desarrolladores han exigido un formato normalizado; AsyncAPI es, pues, la respuesta propuesta para casi todo lo que depende de una arquitectura basada en mensajes.
        • (Dicho esto, un estándar es tan bueno como aquellos que lo aceptan. Para ello, AsyncAPI tiene varios nombres grandes que respaldan su uso e implementación. AsyncAPI es utilizado por gigantes de la industria como Salesforce, MuleSoft, Slack y Tibco con gran eficacia para impulsar la documentación de sus sistemas basados en mensajes.)
        • Independientemente del AsyncAPI, tener un estándar para este tipo de API tiene muchas ventajas por ejemplo:
          • es más fácil de crear herramientas reutilizables
          • y es más fácil de integrar codigo de grupos diferentes
        • ✅ Los beneficios particulares en nuestro caso :
          • No hace cumplir ningún protocolo en particular. AsyncAPI puede evitar un estándar unificado sin tener que desarrollar una amplia variedad de integraciones y sistemas de portabilidad; en gran medida, debería funcionar de manera inmediata.
          • Al definir una relación contractual muy específica entre los elementos de la API, los desarrolladores pueden controlar y gestionar con mayor precisión el flujo de estos mensajes y su resultado.
            • (Uno de los problemas de las messaging API es que los programadores no necesariamente saben quién usa qué cuando hay varios equipos y así que a veces hacen cambios con la esperanza de no romper nada. Mientras que con AsyncAPI es posible saber quién está usando qué.)
          • Está diseñado para apoyar la legibilidad de la máquina y la legibilidad humana. Las APIs pueden ser descritas usando JSON o YAML, y las interfaces gráficas pueden ser integradas con la experiencia de AsyncAPI para una mayor facilidad de uso. Tiene sentido desde el punto de vista comercial, y también desde el punto de vista de los programadores.
        • ❌En cuanto a los puntos débiles siempre habrá normas que compitan entre sí y soluciones adicionales, específicas para cada caso. En el caso de AsyncApi, su problema principal es que es joven ya que sólo tiene dos años. Por lo tanto, hay muchos puntos que aún no han sido abordados. y además es desconocido para una parte del público objetivo.
      • Conclusiones
        • Después de haber investigado esta tecnología, ahora tenemos mucho más conocimiento sobre las APIs, en concreto, de las asíncronas.
        • Ahora sabemos en qué proyectos es mejor el uso de un tipo u otro dependiendo de si quieres respuesta inmediata (síncrona) o no (asíncrona).
        • Además hemos aprendido que es la única API asíncrona con una documentación bien establecida que además es Open Source
        • Gracias a este estudio sabemos que podremos utilizar en un futuro próximo AsyncAPI para APIs que no necesitan de una respuesta inmediata.

P16_AMQP - Advanced Message Queuing Protocol

  • Introducció
    • AMQP estableix un conjunt de regles i una sintaxis per la comunicació entre dos o més participants, permetent una transmissió sòlida de dades amb exactitud i seguretat.
    • Actualment hi ha diferents versions usant-se alhora ja que es basa en l’estàndard de codi obert i el seu ús és gratuït. Algunes empreses que en fan ús:
  • Història
    • Pel sector financer, la transmissió d’informació és fonamental. Una empresa de targetes de crèdit o una borsa de valors intercanvia milers de missatges per segon. Si un missatge arriba tard o no arriba, pot sortir molt car.
    • Per tant, John O’hara, director del projecte en aquell moments, com que no hi havia cap producte comercial capaç de complir amb els requisits satisfactòriament, va desenvolupar un protocol propi.
    • Es va començar a desenvolupar l’any 2003 per JPMorgan Chase, un banc nord-americà, i socis d’empreses tecnològiques com Cisco.
  • Característiques
    • Incorpora esquemes de control de flux. Equilibri entre eficiencia, flexibilitat i interoperabilitat.
    • Confiabilitat en l’intercanvi de missatges.
    • Flexible. Admet diverses tipologies (client-client, client-participant, participant-participant).
    • Independent del model del participant. (no requereix compilació)
    • Seguretat
  • Funcionament
    1. El producer publica un missatge amb una routing-key a un exchange.
    1. L’exchange rep el missatge i l’envía a una (o més) cues.
    1. Els bindings defineixen les possibles queues on pot anar el missatge.
    1. Les queues guarden el missatge fins que un i només un consumer l’agafa o el consumeix.
    1. El consumer rep el missatge i pren les accions necessàries.

    • Tipus d’exchange
      • Segons el tipus d’exchange el missatge s’envia a les cues on...
        • Direct: El routing-key és exactament igual al binding-key.
        • Topic: el routing-key compleix el patró del binding-key.
        • Fanout: Totes les cues enllaçades a l’exchange.
  • Protocols similars
    • MQTT (Message Queue Telemetry Transport)
      • Més simple i fàcil d’implantar que AMQP
      • Potent en transmissió i rebuda de missatges
      • Ús de pocs recursos
      • Flexibilitat en implantació de seguretat
    • STOMP (Simple/Streaming Text Oriented Messaging Protocol)
      • Missatges basats en text (JSON, XML)
      • Permet actualitzar webs i aplicacions en temps real
    • RabbitMQ
      • Agent de missatges (broker) multiprotocol
      • Comunicació entre diverses aplicacions
      • Protocols http
  • ✅ Avantatges
    • És fàcil de mantenir i fàcil d’escalar. (podent implementar diversos microserveis)
    • Permet tenir clients en diversos llenguatges de programació (no restricció)
    • Garanteix l’entrega de missatges.
  • ❌Inconvenients
    • Pot afegir una complexitat addicional al projecte.
    • Les últimes versions(1.0 i 0-9-1) no son retrocompatibles entre elles, aixo disminueix el valor previst de disposar d'un protocol de missatge interoperable.
    • Protocol asíncron (no en temps real)
  • Conclusions
    • L’AMQP és un protocol de missatgeria robust que transmet els missatges a nivell de cable, els quals són octets. Per tant permet que des de qualsevol eina si es segueix el protocol establert, es puguin tractar i interpretar, i així evitar feina i incompatibilitats des de qualsevol altre aplicació o servei.
    • La temàtica del treball ha sigut molt interessant i era un tema el qual desconeixiem els integrants del grup abans de fer el projecte. Entendre com es comuniquen les aplicacions és una part essencial per poder-les desenvolupar.

    • Protocol binari
    • Multiplexing (2 o més senyals pel mateix canal)
    • Compressió de capçaleres ( + eliminació de info redundant)
    • Server Push (no haver d'esperar, intentar endevinar resposta i fer preload)
  • HTTP/2.0: Funcionament intern (16nanoBytes max)
  • ✅ HTTP/2.0: Punts forts
    • Backward Compatibility (1.1 normalment)
    • Seguretat
